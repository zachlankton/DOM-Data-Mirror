    • Introduction

    DOM Data Mirror (dData for short) is an experiment created in an attempt to accomplish a few tasks...

    • Decrease Code Required to create Basic Web Apps
    • Extend HTML, instead of abstracting over it
    • Reduce or eliminate controller code
    • Reduce or eliminate the need for frameworks

    dData, by itself, doesn't necessarily accomplish all of these things. However, dData is extendable and included in the code base are some extensions that start pushing dData in that direction.

    By writing some simple extensions you can considerably reduce the amount of boilerplate required in your app. We will show how this can be done in the examples to follow.

    Installation

    Installing dData is as simple as including the script in the head tag of your project. It is written using vanilla javascript so there are no dependencies.

    Basic Usage

    To Understand what dData does, let's start a simple vanilla js example:

    The d-data attribute may be used on any native html element.

    This attribute extends the element to map values contained inside child elements that have the name attribute.

    The intended effect is to have the DOM mirror the data in the object and vice versa.

    <div d-data name="test"></div> === dData.test // true but not in the sense that dData.test just contains a reference to the Element... this is where some abstraction comes in.

    In this case, the html div code above is the same thing as var test = {}; This is what the d-data attribute is doing behind the scenes. It is a lot like data binding to a template, but in this case the template is generating it's own object to be used elsewhere in the program.

    If the values in the object are updated then the html values (or innerHTML) will be updated and inversely If the DOM is updated, either by user input or another library manipulating the DOM, the object will reflect those changes.

    dData Elements require a name attribute with a value assigned. Just like <input /> tags typically do. The name attribute value is used as the key name in the dData[parentName] object that the values get stored in.

    Child Elements of dData do not require a name attribute, but any element with the name attribute will be mapped to a key inside of the dData[parentName][valueName] = value object. Let's look at some code to hopefully clear up what is meant by all of this...

    Overloading

    What happens if you start with an empty root dData element and then assign data to it as we have in all the previous examples? One, might first assume that it just gets discarded. This could be a problem if you already have an object with data that needs to stay intact, but you really only want to display certain values from it in the html.

    You could manually create hidden input types <input type="hidden" name="someName" > as a placeholder for data that you do not want displayed and in some cases you may still want to do it this way, but you don't have to.
