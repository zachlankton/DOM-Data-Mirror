    Introduction

    DOM Data Mirror (dData for short) is an experiment created in an attempt to accomplish a few tasks...

    • Decrease Code Required to create Basic Web Apps
    • Extend HTML, instead of abstracting over it
    • Reduce or eliminate controller code
    • Reduce or eliminate the need for frameworks

    dData, by itself, doesn't necessarily accomplish all of these things. However, dData is extendable and included in the code base are some extensions that start pushing dData in that direction.

    By writing some simple extensions you can considerably reduce the amount of boilerplate required in your app. We will show how this can be done in the examples to follow.

    Installation

    Installing dData is as simple as including the script in the head tag of your project. It is written using vanilla javascript so there are no dependencies.

    d-data Explained

    The d-data attribute may be used on any native html element.

    This attribute extends the element to map values contained inside d-data element's children that have the name attribute.

    The intended effect is to have the DOM mirror the data in the object and vice versa.

    <div d-data name="test"></div> === dData.test // true but not in the sense that dData.test just contains a reference to the Element... this is where some abstraction comes in.

    In this case, the html div code above is the same thing as dData.test = {}; This is what the d-data attribute is doing behind the scenes. It is a lot like data binding to a template, but in this case the template is generating it's own object to be used elsewhere in the program.

    Note: dData is a global object that is the default "scope" where your dData objects will be stored. This global object also contains some methods for working with dData elements. You can set your own scope using the scope attribute. See Here for more info about scope

    If the values in the object are updated then the html values (or innerHTML) will be updated and inversely If the DOM is updated, either by user input or another library manipulating the DOM, the object will reflect those changes.

    dData Elements require a name attribute with a value assigned. Just like <input /> tags typically do. The name attribute value is used as the key name in the dData[parentName] object that the values get stored in.

    Child elements of a d-data element do not require a name attribute, but any element with the name attribute will be mapped to a key inside of the dData[parentName][valueName] = value object.

    Glossary

    Term Definition
    d-data element Any element that has a d-data attribute.
    d-data child element These are also "d-data elements" because they have a d-data attribute. The differnentiator here is that they are descendants of another d-data element. These are explained in the Arrays (Child Elements) section.
    d-data element children Any descendant of a d-data element that has a name attribute.
    d-data object The javascript object that gets and set values on d-data elements.

    Arrays (Child Elements)

    dData Elements can be nested infinitely... the root d-data element is an Object, and it inherits its scope from the global environment.

    All d-data child elements are arrays... this allows for them and their content to be duplicated.

    dData child elements inherit their scope from their parent d-data element.

    Defining a d-data element to d-data child element structure is identical to the structure of a javascript object and a lot like writing an object literal.

    Overloading

    What happens if you start with an empty root dData element and then assign data to it as we have in all the previous examples? One might first assume that it just gets discarded. This could be a problem if you already have an object with data that needs to stay intact, but you really only want to display certain values from it in the html.

    You could manually create hidden input types <input type="hidden" name="someName" > as a placeholder for data that you do not want displayed and in some cases you may still want to do it this way, but you don't have to.

    Scope

    If you are familiar with Angular, Vue, React, and other JS frameworks, then you are probably familiar with the concept of scope. Scope helps keep code modular and separated from other code. Global Variables have a higher chance of overwriting something that already exists, or being overwritten in the future by some other code that comes along.

    When building simple applications, if less than a few people are contributing and can keep track of the names of their dData objects and everyone elses, using the default dData[objectName] scope should work just fine.

    When building larger, more complex, applications with more contributors, making use of different scopes becomes imperative to keeping code organized and helping to prevent really strange bugs that are very difficult to track down. You and you're team will need to decide the best approach for your application.

    To use your own custom scope with dData, you can use the scope attribute on the root dData Element. This will not work on d-data child elements. d-data child elements inherit their scope directly from their parent d-data element.
